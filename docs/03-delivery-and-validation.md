# UI Bridge：macOS 实施、测试与确认项

状态：macOS 核心、客户端写入闭环和内部名称迁移均已验收

## 1. 推荐项目结构

```text
ui-bridge/
├── app/                         # macOS 常驻服务 App
├── packages/
│   ├── core/                    # 控件、窗口、快照、动作和验证
│   ├── server/                  # 本地 HTTP 与会话
│   ├── mcp-adapter/             # MCP 入口
│   └── protocol/                # JSON Schema 与生成模型
├── skills/
│   └── macos-ui-control/        # Cursor/WorkBuddy/Codex Skill
├── fixtures/                    # 脱敏控件树和截图测试材料
├── tests/
│   ├── unit/
│   ├── contract/
│   ├── integration/
│   └── app-regression/
├── scripts/
├── docs/
└── Package.swift
```

当前代码、App、MCP、Skill、客户端接入和验收材料均已落地；本结构是设计视图，
实际 Swift 包结构以仓库 `Sources/` 为准。

## 2. 开发阶段

### 阶段 0：通用底层探针

目标：先验证四类代表应用是否能由同一套通用读取器处理，不写应用专属逻辑。

- TextEdit：标准输入框和文本写入。
- Finder：窗口、列表、选择和滚动。
- 企业微信：虚拟列表、联系人行和聊天输入框。
- 一个 Electron 应用：网页式控件与结构不完整判断。

探针代码直接进入通用核心，不做一次性脚本。若企业微信需要特殊读取关系，只能
以“虚拟表格/可见行”的通用能力实现。

### 阶段 1：通用只读核心与端口

- App 和窗口发现。
- 完整辅助功能读取。
- 单窗口截图。
- 快照与控件句柄。
- 常驻服务 App、启动/停止/状态命令。
- 权限检查与诊断命令。
- HTTP 只读接口和基础令牌。

### 阶段 2：动作与验证

- 语义点击、选择和写值。
- 指定进程的键盘和滚动操作。
- 坐标换算和窗口内点击。
- 新快照验证。
- 过期快照、窗口变化和应用重启处理。

### 阶段 3：MCP、Skill 与客户端

- MCP stdio。
- MCP Streamable HTTP。
- Cursor 配置。
- WorkBuddy 接入实测。
- 可安装、可触发、经过真实任务验证的通用 Skill。

### 阶段 4：第一轮安全收口

- 命令行生成和轮换客户端令牌。
- 应用级允许/禁止规则。
- 高影响动作必须在常驻 App 中按具体目标再次确认；拒绝、关闭或超时均不执行，允许只对当前请求生效。
- 紧急停止。
- 隐私遮盖、诊断摘要和用户主动导出的脱敏 JSON 诊断报告；完整审计日志包尚未实现。
- 稳定开发签名和本地安装脚本。

### 阶段 5：通用回归与第一轮交付

- Finder、系统设置、文本编辑等原生应用。
- 企业微信。
- Chrome 外壳与 Electron 应用。
- 普通微信只做能力评估，不加入专属实现。
- 兼容性报告与已知限制。
- 端到端 MCP、HTTP 和 Skill 验收。

## 3. 测试策略

### 单元测试

- 辅助功能属性转换。
- 树遍历、循环引用和节点上限。
- 控件 fingerprint 与句柄失效。
- Retina 与多屏坐标换算。
- 风险分类和确认门槛。
- 敏感字段遮盖。

### 契约测试

- JSON Schema 与 Swift/TypeScript 模型一致。
- MCP 与 HTTP 对相同请求返回等价结果。
- 所有错误码都具有稳定字段。
- 旧客户端遇到新增字段不崩溃。

### 集成测试

建立一个自有测试 App，包含：

- 标准按钮、输入框、表格和树。
- 虚拟滚动列表。
- 动态重建控件。
- 自绘区域。
- 弹窗、菜单和多窗口。
- 可模拟后台事件丢失。

测试 App 提供确定的状态接口，使自动测试能判断操作是否真正生效。

### 真实应用回归

每个版本至少测试：

- Finder：导航和文件选择，不做删除。
- TextEdit：后台输入和内容验证。
- 系统设置：只读与普通导航。
- 企业微信：搜索联系人、选择聊天、填写草稿，不默认发送。
- Chrome：地址栏与标签页外壳；网页内容优先交给浏览器通道。
- 普通微信：记录可读取范围和前台需求。

真实应用测试默认不执行外部发送。发送测试使用专用测试联系人，并在动作前确认。

## 4. 关键场景验收

### 场景 A：通用控件操作

在自有测试 App 和 TextEdit 中完成窗口发现、快照、控件查找、输入、点击和结果
验证，证明核心能力不依赖第三方应用。

此外，Cursor 与 WorkBuddy 必须分别从真实客户端发起一次隔离 TextEdit 写入。每次都要
先检查写入方案，再执行写入，并使用动作返回的新快照精确回读。只看到工具、只读快照、
或客户端口头报告成功都不算通过；目标文稿中的实际内容也必须一致。测试使用新建且未保存
的空白文稿，不得复用用户已有文稿。

若 WorkBuddy 没有展示完整写入工具，允许使用已安装 App 的本地 `call` 入口完成同一套
检查、写入和回读。该入口自行读取本机凭据，提示词和命令中不得出现令牌。验收结束还要
检查工作区，清除 WorkBuddy 自行生成且只属于本次测试的记忆文件。

### 场景 B：Finder 列表

按语义选择指定文件或目录并验证选中状态；窗口移动后旧快照必须失效。

### 场景 C：企业微信虚拟列表

1. 后台读取企业微信。
2. 搜索指定测试联系人。
3. 返回唯一匹配行及其文本。
4. 按行选择。
5. 验证聊天标题。
6. 找到输入框。
7. 写入草稿。
8. 验证草稿，不发送。

要求：不猜坐标、不误选、不抢前台。

企业微信只作为复杂虚拟列表样本，核心代码不得出现应用名称、联系人名称或固定
坐标。

### 场景 D：窗口变化

1. 获取快照。
2. 移动或缩放窗口。
3. 使用旧句柄执行动作。
4. 服务返回快照过期。
5. 重新读取后成功操作。

要求：旧坐标绝不能静默执行。

### 场景 E：后台失败

1. 目标应用丢弃后台输入。
2. 服务验证不到变化。
3. 返回需要前台及具体证据。
4. 未确认前不切前台。
5. 确认后只对受阻动作短暂切前台并恢复。

### 场景 F：外部发送

1. 准备目标、内容和输入框。
2. 返回待确认摘要。
3. 在常驻 App 中确认当前目标和具体动作。
4. 执行一次发送。
5. 验证消息出现在正确聊天。
6. 同一确认不能用于第二次动作。

## 5. 性能目标

第一版建议目标：

- 列出应用：200 毫秒内。
- 普通窗口完整快照：1 秒内。
- 2,000 节点复杂窗口快照：2 秒内。
- 不含加载的语义动作：500 毫秒内返回初步结果。
- 常驻空闲内存：150 MB 内。
- 空闲 CPU：接近 0%。

这些目标在技术探针后依据真实数据调整。

## 6. 质量门槛

以下是最初设定的完整门槛。已经取得的实际证据与尚未完成项以
[`04-current-status.md`](04-current-status.md) 为准，不能把设计目标当作已验证事实。

第一轮交付前必须满足：

- 所有核心模块有单元测试。
- 协议契约测试通过。
- 关键场景 A 至 F 中除真实发送外全部通过；发送流程通过模拟目标验证确认门槛。
- 四类代表应用均完成端到端 MCP 调用，结果与直接 HTTP 调用一致。
- 企业微信连续 20 次联系人选择无误点，且核心代码没有应用名称和固定坐标。
- 多屏和 Retina 坐标测试通过。
- 权限撤销、应用退出、窗口关闭和客户端断开不会卡死。
- 紧急停止可在 1 秒内阻止新动作。
- 日志审查未发现密码和完整聊天正文。
- Cursor 与 WorkBuddy 都能通过 MCP 对隔离 TextEdit 文稿完成真实写入，并从最新快照精确回读。
- 通用 Skill 通过格式检查和四类代表任务验证。

进入公开发布前另需完成正式签名、公证、自动更新和可视化权限管理，这些不属于
第一轮。

## 7. 开发纪律

- 每完成一个动作能力，先在自有测试 App 验证，再测真实应用。
- 不用“系统调用成功”替代界面结果验证。
- 不为单个应用加入未经记录的固定坐标。
- 失败两种不同路线后停止，先定位原因。
- 每个兼容性修复保留脱敏回归材料。
- 所有高影响动作测试使用专用账号和测试联系人。

## 8. 第一轮交付结果

第一轮已交付一个可用的 macOS 通用核心版：

- 可运行的常驻服务 App，通过本地端口工作。
- `start`、`stop`、`status`、`permissions`、`token`、`call` 命令。
- 任意应用和窗口发现。
- 通用控件树、单窗口截图、结构质量判断和快照句柄。
- 通用点击、选择、写值、输入、按键、滚动和窗口内坐标操作。
- 操作后验证、旧快照保护、后台失败和前台升级结果。
- 本地 HTTP 接口、MCP stdio 与 MCP Streamable HTTP。
- Cursor 接入配置。
- WorkBuddy 可用入口实测。
- 可安装的通用 Skill。
- TextEdit、Finder、企业微信和 Electron 四类真实应用回归报告。

当前仍不做：

- 完整设置 UI。
- 自动更新。
- 正式安装包。
- 对外发送。
- Windows。
- 应用专属脚本和固定坐标适配。
- 云端视觉模型。

## 9. 需要用户确认的决策

### 决策 1：产品定位

推荐：先做个人本机工具，架构保留未来发布能力。

影响：个人工具可以先不做团队账号、云服务和复杂许可证，显著缩短第一版周期。

### 决策 2：最低系统版本

推荐：macOS 14.4。

影响：可以使用现代窗口捕获和并发能力；更低版本需要额外兼容投入。

### 决策 3：第一优先应用

推荐用 TextEdit、Finder、企业微信、Electron 四类样本并行验收同一个通用核心，
不设置“企业微信优先开发”的产品路线。

影响：核心能力从第一天保持通用；企业微信负责暴露虚拟列表问题，而不是决定架构。

### 决策 4：客户端入口

推荐：MCP stdio 与本地 HTTP 同时设计，先实现 HTTP 核心，再接 MCP。

影响：WorkBuddy 即使不支持 MCP，也可通过本地接口接入。

### 决策 5：视觉识别

推荐：第一版不内置云端视觉模型，只保留扩展接口。

影响：先把结构读取和可靠操作做好，避免截图外发、费用和隐私问题扩大范围。

### 决策 6：项目名称

显示名称和 MCP 连接名分别为 `UI Bridge` 与 `ui-bridge`。macOS 安装包内部程序名、系统身份、启动项和本地状态目录全部使用新名称。

影响：客户端配置会自动备份并把旧连接项迁移到新名称；旧应用和启动项会被清理，本地状态会迁移。由于系统身份变化，升级后需要重新授予辅助功能和录屏权限。端口保持不变。
