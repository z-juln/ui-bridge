# UI Bridge：Web 接入方向（待细化）

状态：方向已确认，尚未进入实现；本文件只固定边界，不提前承诺技术方案。

## 1. 目标

让 UI Bridge 使用同一套客户端入口、安全规则、实时活动和结果验证，处理三类界面目标：

- `native_app`：当前已经支持的 macOS 原生应用窗口。
- `browser_page`：浏览器中的普通网页和标签页。
- `embedded_webview`：桌面应用内部嵌入的网页区域。

对 Agent 来说，先发现目标和可用能力，再选择读取与执行通道；不要求 Agent 根据应用名称猜测路线。

## 2. 保持统一的部分

- 本地 HTTP、MCP 和 Skill 入口。
- 客户端身份、连接状态和紧急停止。
- 只读、普通写入、前台升级和危险操作确认规则。
- 当前状态、实时画面、客户端到目标的映射和事件记录。
- 每次写操作后重新读取并验证结果。
- 敏感字段遮盖、诊断导出和本机回环限制。

## 3. 分开实现的部分

| 目标 | 优先读取方式 | 优先操作方式 | 画面补充 |
| --- | --- | --- | --- |
| 原生应用 | macOS 辅助功能树 | 控件公开动作与指定进程输入 | 单窗口画面 |
| 浏览器页面 | 页面结构与浏览器通道 | 页面控件动作 | 标签页或窗口画面 |
| 内嵌 WebView | 可用的页面结构；不可用时退回原生可访问树 | 页面通道或宿主应用控件 | WebView 所在窗口区域 |

任何通道都不能因为底层调用返回成功就报告任务成功，必须从新状态得到证据。

## 4. 计划阶段

### 阶段 A：统一目标模型

- 把当前应用和窗口模型扩展为界面目标，不破坏现有 macOS 客户端。
- 返回目标类型、可用能力、所属应用、窗口或标签页关系。
- 设置页和实时操控页能同时展示不同类型目标。

### 阶段 B：浏览器页面只读

- 发现浏览器、窗口和标签页。
- 读取页面结构、可见区域和基本状态。
- 处理页面导航后旧目标失效。
- 不执行登录、提交、上传或下载。

### 阶段 C：浏览器页面安全操作

- 点击、填写、选择、滚动和导航。
- 写入后重新读取页面状态。
- 上传、提交、外部发送、购买和权限请求继续走危险操作确认。

### 阶段 D：内嵌 WebView

- 识别宿主应用中的 WebView 区域。
- 在页面通道和原生可访问树之间选择可靠路线。
- 将 WebView 坐标统一到宿主窗口，窗口变化后旧坐标立即失效。

### 阶段 E：跨平台

- 在统一目标和协议稳定后评估 Windows 原生应用。
- Windows 共享上层协议和安全模型，不复用 macOS 的底层实现。

## 5. 待定问题

以下问题在开始 Web 实现前单独做技术验证，本轮不做决定：

- 浏览器连接采用扩展、调试协议、浏览器内嵌能力，还是组合方案。
- 如何在不接管用户浏览器的前提下复用现有登录状态。
- 内嵌 WebView 如何发现调试入口，以及入口不可用时的降级边界。
- 页面结构、可访问树和截图发生冲突时，以什么证据判断结果。
- 跨域页面、弹窗、下载、文件选择器和浏览器权限提示如何建模。
- 浏览器页面画面如何进入实时操控页，同时避免持续保存页面内容。
- Web 会话如何与客户端身份、窗口级并发和危险操作确认绑定。

## 6. 不变的安全边界

- 不绕过登录、验证码、安全警告、浏览器权限或网站自身限制。
- 不默认导出 Cookie、令牌、历史记录或完整页面正文。
- 不允许网页内容向 UI Bridge 下达扩大权限的指令。
- 不把“已点击”当作“已提交成功”，也不在结果不明确时自动重复。
- 当前 macOS 原生能力不得为了 Web 接入而降低隔离和验证要求。
