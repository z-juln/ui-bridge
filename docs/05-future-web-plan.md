# UI Bridge：Web Bridge 计划

状态：产品方向已确认，尚未进入实现。目标是至少达到 Codex 内嵌浏览器的使用体验，
并在控件理解、经验复用、结果验证和可见安全状态上更进一步。

## 1. 产品目标

UI Bridge 使用同一套客户端入口、安全规则、实时活动和结果验证，处理三类界面目标：

- `native_app`：当前已支持的 macOS 原生应用窗口。
- `browser_page`：UI Bridge 内嵌浏览器或用户现有 Chrome 中的网页和标签页。
- `embedded_webview`：桌面应用内部嵌入的网页区域。

Agent 只表达要读取或操作的目标，不根据应用名称猜测底层路线。Bridge 根据目标类型、
当前能力和安全策略选择通道。

## 2. 两种浏览器模式

### UI Bridge 管理的内嵌浏览器

- 内嵌 Chromium，而不是分发 Google Chrome 本体。
- 每个浏览器用户空间独立保存 Cookie、密码、历史和站点权限。
- 支持标签页、弹窗、下载、文件选择器和浏览器权限提示的统一建模。
- 页面读取和输入发生在浏览器内部，不移动系统鼠标，也不要求窗口位于最前方。
- 页面和标签页画面进入现有“实时操控”界面。

CEF/Chromium 是第一实现候选，但进入正式开发前必须先验证 macOS 打包、更新、资料导入、
密码库和 Windows 复用成本。不能假设浏览器内核会自动提供完整 Chrome 密码管理能力。

### 连接用户现有 Chrome

- 通过 Chrome 扩展和本机连接发现用户明确授权的标签页。
- 复用现有 Chrome 的账号、扩展和密码管理器，不复制资料即可操作。
- 扩展按网站申请权限，不默认获得全部浏览记录或全部网站内容。
- 连接现有 Chrome 与资料导入是两个独立能力，用户可以分别开启。

## 3. 读取、定位和操作

网页目标同时使用三类信息：

1. DOM 和布局：元素、属性、iframe、Shadow DOM、可见范围和页面坐标。
2. 页面无障碍树：角色、名称、状态、可用动作和语义关系。
3. 实时画面：补充 Canvas、自绘控件和结构缺失区域。

优先按角色、标签和稳定属性定位控件，再从当前布局取得位置。固定坐标、DOM 顺序和临时
节点编号不能作为长期经验。每次导航、刷新或主要布局变化后，旧页面快照立即失效。

动作优先通过浏览器内部输入完成。执行后必须重新读取 URL、页面结构、目标状态和可见反馈；
不能因为点击或脚本调用返回成功就报告任务完成。

操作经验由 Bridge 从已验证任务中自动沉淀、复测、升降级和淘汰，不要求用户教学或 Agent 管理，统一由
[`06-experience-library-plan.md`](06-experience-library-plan.md) 定义。

## 4. Chrome 密码与 Cookie 导入

导入体验与 Codex 保持一致：

- 识别本机 Chrome 及其多个用户。
- 用户选择来源 Chrome 用户和目标 UI Bridge 浏览器空间。
- 导入前检测 Chrome 是否完全退出，并提供明确处理提示。
- 密码和 Cookie 分别提供开关。
- 展示导入进度、成功数量、失败原因和重试入口。
- 支持删除整个浏览器空间及其中的登录资料。

内部边界：

- 密码写入内嵌浏览器的加密密码库，Cookie 写入目标浏览器空间。
- 密码、Cookie 和令牌原文不进入 MCP、HTTP、Skill、经验库、诊断或普通日志。
- Agent 只能请求使用浏览器自动填充或已有登录状态，不能读取或复制原文。
- 不把 Chrome 用户目录直接作为 UI Bridge 的运行目录，也不在 Chrome 运行时复制数据库。
- macOS 先完成真实导入验证；Windows 必须单独处理系统加密与浏览器保护机制。

## 5. 统一安全和可见反馈

- 内嵌浏览器、现有 Chrome 和原生 App 共用客户端身份、紧急停止和危险操作确认。
- 登录、验证码、安全警告和浏览器权限不能被绕过。
- 发送、提交、上传、下载、购买、删除和权限变更按影响请求确认。
- 网页正文一律视为不可信输入，不能修改 Bridge 规则或扩大权限。
- 实时操控页展示客户端、浏览器空间、标签页、当前动作、使用的经验和验证结果。
- 页面画面只保留内存中的最新帧，不默认长期保存。

## 6. 实施阶段

### 阶段 A：技术验证和统一目标

- 验证 Chromium 嵌入、签名、辅助进程、升级和资源占用。
- 验证 Chrome 多用户发现，以及密码和 Cookie 的真实本地导入。
- 扩展当前应用/窗口模型，使其能表达浏览器、浏览器空间、窗口和标签页。
- 保持现有 macOS MCP 工具兼容。

### 阶段 B：内嵌浏览器只读闭环

- 浏览器窗口、地址栏、标签页和浏览器空间管理。
- 页面结构、无障碍树、可见画面和导航失效机制。
- 实时操控页展示真实标签页画面和客户端映射。
- 不执行登录、提交、上传或下载。

### 阶段 C：安全操作闭环

- 点击、填写、选择、滚动、导航和操作后复查。
- 弹窗、跨域 iframe、Shadow DOM、文件选择器和浏览器权限处理。
- 提交、发送、下载、上传和购买接入现有危险操作确认。

### 阶段 D：资料导入和现有 Chrome

- 实现与 Codex 一致的 Chrome 用户、密码和 Cookie 导入界面。
- 增加 Chrome 扩展和本机连接，支持复用现有浏览器登录环境。
- 资料导入、扩展授权和浏览器空间均提供可查看、撤销和删除入口。

### 阶段 E：经验库与内嵌 WebView

- 网页进入统一操作经验库，支持自动生成、影子复测、升降级和失效重建。
- 识别宿主应用中的 WebView，在页面通道与原生可访问树之间选择可靠路线。
- 将 WebView 坐标统一到宿主窗口，窗口变化后旧坐标立即失效。

### 阶段 F：Windows

- 浏览器上层目标、安全规则、经验格式和客户端接口跨平台复用。
- Windows 原生应用使用独立底层实现，不复用 macOS 系统调用。

## 7. 第一轮完成标准

- 用户能创建独立浏览器空间并打开多个标签页。
- Cursor 或 WorkBuddy 能只读页面、按控件操作且不移动系统鼠标。
- 页面操作后有新状态证据，页面变化后旧快照不能继续使用。
- UI Bridge 能实时显示正在操控的浏览器空间和标签页。
- Chrome 多用户、密码和 Cookie 导入完成真实验收，原文不会出现在任何客户端返回中。
- 登录、提交、上传、下载、购买和权限请求符合统一安全规则。
