# 实现 WorkBuddy Cursor 真实写入闭环 - 发现记录

本文件记录任务执行中形成的判断、事实和技术决策。它不是审查报告；阻塞性问题请写入 `review.md`。

## 研究发现

### WorkBuddy 写工具暴露不完整

- 背景：WorkBuddy 可以读取 Bridge，但首轮真实写入时没有展示 `plan_check` 和 `action_run`。
- 发现：已安装 App 内增加自行读取本机凭据的 `call` 入口后，WorkBuddy 可以逐条完成同一协议链路，无需在提示词或仓库文件中保存令牌。
- 影响：WorkBuddy 默认优先 MCP；工具不完整时使用安全本地调用，不再用临时脚本拼接 HTTP。
- 后续：接入说明已记录完整参数和限制。

### HTTP 请求正文可能分段到达

- 背景：本地调用使用系统网络客户端时，服务偶发只收到请求头，较大 JSON 正文随后到达。
- 发现：旧服务只读取一次连接数据，会把分段请求误判为坏请求；按正文长度累积完整请求后真实写入通过。
- 影响：所有 HTTP 与本地调用对分段正文稳定，仍保留 1 MB 上限。
- 后续：已用真实 WorkBuddy 写入链路验证。

### 命令错误不应触发系统崩溃弹窗

- 背景：无效 `call` 参数作为未处理错误退出时，macOS 显示“意外退出”。
- 发现：在命令入口统一收口错误、打印简短说明并返回非零状态后，不再生成新崩溃报告。
- 影响：Agent 参数错误不会干扰用户桌面。
- 后续：已加入安装后负向检查。

## 技术决策

| 决策 | 选择 | 原因 | 替代方案 | 状态 |
| --- | --- | --- | --- | --- |
| WorkBuddy 回退入口 | 已安装 App 的 `call` 子命令 | 不暴露令牌、复用同一服务和协议、可逐条授权 | 临时 Python 脚本、把令牌写入提示词 | accepted |
| 写后确认 | 动作返回的新快照 + 独立界面核对 | 排除客户端口头成功与旧快照假阳性 | 只看命令退出码 | accepted |
